## 高价值做法

### 1. 先用参考 Wiki 的导航结构做系统分层，再决定抓取策略

这次有效的第一步不是立刻写 scraper，而是先确认 Wiki 主导航已经天然暴露了系统边界：

- 单位
- 命运卡
- 房间
- 技能
- 遗物
- 装备
- 折磨
- 模式与额外玩法

这样做的价值是，后续抓取不是“见页就抄”，而是围绕开发侧的数据模型来组织材料。

### 2. 对“参考资料整理”类任务，交付物契约要尽早压实到本地资产

第一版其实已经不错：

- 人读版框架说明
- JSON Schema 草案
- Canvas 图谱

但用户真正要的是“后续开发的物料”，所以仅有说明和链接不够。第二版把交付收敛成：

- `tools/scrape_dungeon_maker_configs.py`
- `data/config/*.json`
- `data/config/README.md`
- `data/raw/dungeon_maker_wiki/*.txt`

这个转向很关键。对外部资料整理类工作，真正可复用的不是总结文本，而是“可复跑的生成路径 + 本地结构化产物”。

### 3. 批量抓取前先验证 1 到 2 条抽取路径，能显著减少返工

这条线程中有一个很好的中间动作：

- 先验证怪物星级分类页能否稳定枚举条目。
- 发现英雄类分类页并不按同一模式返回结果。
- 立即改成“候选页面包含 `PageHero` 模板”作为回退识别策略。

这说明对半结构化 Wiki 抓取，不要假设所有栏目都遵循同一模板。先做小样验证，比直接全量抓完再返工更稳。

### 4. 说明文档、机器配置、原始缓存、解析校验最好一起交付

第二版交付里最值得复用的是四件套同时存在：

- 人读说明，方便理解系统设计。
- 机器可用 JSON，方便直接接开发。
- 原始 wikitext 缓存，方便以后重跑或修 parser。
- 全量 JSON 解析校验，确保不是“写出来但不可用”。

这比只给 Markdown 摘要更接近工程资产。

### 5. 工具链组合很实用：网页确认结构，Shell 跑批量，补丁落脚本，长任务轮询输出

这条线程的工具使用模式比较清晰：

- 先用网页访问确认参考 Wiki 入口和栏目结构。
- 用 `exec_command` 跑批量探测和抓取。
- 用 `apply_patch` 把 scraper 和说明文件沉淀进工作区。
- 对运行中的抓取脚本用 `write_stdin` 轮询结果，而不是盲等。

如果任务本质是“外部资料 -> 本地可用资产”，这套组合比纯聊天式整理有效得多。

## 暴露出的失败模式

### 1. “整理资料”不等于“生成开发物料”

用户第一句已经提到了“作为后续开发的物料”，但如果没有把交付形态明确写成：

- 本地文件
- 结构化 JSON
- 可复跑脚本

Agent 很容易默认先交“说明文档 + 链接 + 图谱”。这次返工就是因为交付契约在第一轮还不够硬。

### 2. 不同分类页的组织方式可能不一致，不能把首个成功样本直接推广到全站

怪物分类页能直接枚举，不代表英雄、房间、技能都能走同一路径。这个差异如果不先验证，很容易让全量抓取在中段卡住。

## 可复用的 prompt 写法

### 参考资料转开发资产

> 先按参考 Wiki 的导航把系统拆层，再把每一层映射成开发侧可配置的数据模型。交付不要停在说明文档，必须包含本地结构化文件、生成脚本、原始缓存和基本校验结果。

### 半结构化站点抓取

> 不要先全量抓。先验证 1 到 2 条栏目路径能否稳定枚举和解析；如果不同栏目模板不一致，及时切换到模板识别或回退策略，不要手工补样例。

### 交付物契约澄清

> 如果目标是“后续开发物料”，默认交付应是仓库内可直接消费的文件，而不是网页链接、长摘要或单次回答。

## 后续动作

- 以后做“参考产品拆解”时，开头就把最终交付写成具体文件清单，避免先做成分析稿再返工。
- 对 Wiki/知识站点抓取，优先沉淀可复跑脚本和原始缓存，不要只保留解析后的结果。
- 每次全量生成后都保留最小校验动作，至少确认结构化文件全部可解析。
