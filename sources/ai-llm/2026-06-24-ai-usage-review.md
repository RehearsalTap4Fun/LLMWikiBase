## 高价值做法

### 1. 几何/配置 bug 的 prompt，先把“修单点”改成“补不变量”

这次最有效的起手不是“把球往前挪 0.5 米”，而是：

> 先定位切片模版与生成逻辑，确认 `BallPos`、`BallOwner`、`facing` 的约定；先证明问题来自源数据或生成器，而不是 Excel 展示；补一个能卡住重叠问题的校验，再改生成逻辑。

这类 prompt 的好处是，Agent 会自然进入“不变量优先”模式：

- 先抽象成 `BallPos` 必须满足的几何关系。
- 再决定改生成器、测试还是一致性脚本。
- 结果不只修眼前这个 preset，而是给后续新增 preset 也加上护栏。

### 2. TDD 的精神比工具形式更重要

这次两次直接跑 `pytest` 都失败：

- 系统 `python3` 没有 `pytest`
- Codex 自带 Python 也没有 `pytest`

但调试并没有卡死，而是退回到更本质的红灯验证：

- 直接用短 Python 片段导入测试文件。
- 只执行新增的断言函数。
- 先亲眼看到它失败，再继续改生成器。

结论：当本机没有完整测试框架时，不要立刻装环境或跳过验证；先用最小可执行路径证明“测试确实在抓你要修的问题”。

### 3. 需求边界被纠正时，要立即收缩规则而不是硬守原假设

Agent 初版方案一度把“固定球点”的切片改成了反向挪球员，理由是保住点球/角球/任意球的法定球点。用户随后明确了边界：

- 原则上只该改球位置。
- 点球和角球才是固定球点、反向调整持球人。
- 计算结果要做一位小数美化。

这次提醒很有代表性：领域逻辑一旦被用户收口，Agent 最该做的是把规则重新压窄，并同步修测试和一致性检查，而不是只在最终说明里解释偏差。

### 4. 生成器修复后，最好把验证分两层

这次有效的验证链条是：

- 局部断言：新增几何测试，直接跑相关测试函数。
- 全局扫表：跑 `scripts/check_preset_consistency.py`，确认 50 个 preset 都合规。
- 产物再生成：重新导出 `ActivitySoccer_preview.xlsx` / `ActivitySoccerLanguage.xlsx`。

这种分层很实用。局部断言负责回答“我修的是不是这个 bug”，全局审计负责回答“有没有把其他 preset 一起带坏”。

## 暴露出的失败模式

### 环境失败

- 仓库里有 `requirements-dev.txt`，但当前环境没有 `pytest`。
- 连 Codex 桌面自带 Python 运行时也不含 `pytest`。

这说明“仓库声明了依赖”不等于“当前机器可直接跑测试”。

### 认知失败

- 如果用户原句是“球和球员重叠”，默认动作应该是先最小化修改范围。
- 没有用户进一步授权前，不要顺手把相邻对象一起重排，即使那样在几何上更整齐。

## 可复用的 prompt 写法

### 配置/几何类排查

> 先把问题抽成可验证的不变量，证明错误来自源数据/生成器/校验的哪一层；先做一个最小红灯，再改生成逻辑，最后做全局一致性回扫。

### 受限环境下的验证

> 如果本机没有完整测试框架，不要先装环境；优先找最小可执行路径直接调用断言或核心函数，先证明红灯，再做修改。

### 中途纠边界

> 当用户修正规则边界时，把规则、测试、校验脚本一起收窄；不要只改实现，不改护栏。

## 后续动作

- 给常做配置生成的仓库准备一个最小测试入口，至少保证 `pytest` 或等价验证工具随仓可用。
- 把“局部断言 + 全局一致性脚本 + 产物重导出”固化成配置类改动的默认流程。
