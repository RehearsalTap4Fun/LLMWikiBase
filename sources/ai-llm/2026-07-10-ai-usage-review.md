## 关键经验

### 1. 最新信息类问题要先锁“事实边界”，再做表达包装

Codex 的 GPT-5.6 会话里，最有效的一步不是立刻介绍模型，而是先确认“你指的是 OpenAI 的 GPT-5.6 吗”，再按 **截至 2026-07-10** 的官方资料核对模型存在性、预览状态和价格。之后再用 canvas 做对比展示。

可复用点：

- 先写明“按哪个产品理解问题”。
- 对时效强的内容显式写 `截至 YYYY-MM-DD`。
- 先把官方事实核准，再做中文对比页或决策建议。

### 2. 旧审核稿只能提供问题线索，不能替代“最新版对象确认”

K1 策划案审阅里，先做的是确认“当前最新稿”而不是直接沿用昨天的审核结论。这一步避免了最常见的文档类误判：拿旧评论审新稿。

可复用点：

- 先核对主案、派生文档、审核稿的修改时间和 git 状态。
- 审阅结论里区分“仍成立的问题”和“已被新版覆盖的问题”。
- 明确“以当前工作区未提交内容为准”，不要默认上一次审核对象还是当前对象。

### 3. 一旦用户口头拍板了跨章节规则，先记“决策日志”，再批量同步正文

当天最稳的做法出现在 Codex 和 Claude Code 两边：用户补充四邻规则、飞地处理、联盟重建口径后，不是直接散改正文，而是先形成单独的定案记录，再把它同步进主案、配置表和审核稿。

这类任务的本质不是“改一份文档”，而是“维护多份文档的同一个真相源”。

可复用模板：

> 先把本轮定案写成一个短决策表，再逐个同步到主案、派生文档、审核结论，最后跑一次关键词/交叉引用检查。

### 4. 大规格文档修改，应该把“技术假设”和“策划待拍板”分层

MindToDoc 会话里把 `待技术确认项` 和普通 `待拍板项` 分开，是很有价值的结构化动作。这样能避免把程序前置假设误写成策划已经定案的规则。

适用场景：

- 文档里既有玩法规则，也有实现前提。
- 某些条目需要程序先确认地图实例、已有对象类型、兼容边界。

### 5. 讨论产品方向时，先重排“交付物定义”，再决定该不该继续打磨代码

TDGame 会话先把目标从“做一个 React 原型”重排成四个更真实的交付物：战斗规则规格、确定性 sim、可导出的配置、平衡工具链。这个重排直接改变了后续优先级。

可复用点：

- 如果项目最终要被别的宿主/引擎重写，优先沉淀可迁移资产，而不是过度打磨一次性 UI。
- 先问“最终团队真正要接收什么”，再问“当前代码里哪些部分值得继续投入”。

### 6. 设计讨论能直接落到实现时，最有效的方式是“子任务拆分 + 验证闭环”

TDGame 当天不是让一个长会话从头改到底，而是拆成几个明确任务：meta 系统、sim 注入、city UI、App 三态，再补测试、build、dev server、截图验证。

这个模式比笼统地说“把内城做出来”更稳，因为每个子任务都有清晰验收点。

### 7. UI/场景改动不要只看代码，必须走“截图回看”闭环

TDGame 的内城与峡谷场景反复用截图检查默认镜头、房间是否出框、上下两屏是否接缝自然，最后还追到图生图上传接口的真实依赖。对视觉任务来说，`typecheck + build` 远远不够。

可复用点：

- 视觉改动至少要补一个截图或浏览器验证步骤。
- 若问题是“看起来不对”，尽量把反馈变成具体可测条件，例如“一屏内完整看到核心房”和“两张背景合并为恰好两个可视高度”。

### 8. 全局安装第三方技能/插件前，先核对“你信任的是哪个仓库”

SkillsShare 会话里有个很实用的防坑点：用户给出的 fork 仓库，README 却指向另一个上游仓库的 marketplace 安装地址。这里先停下来核对来源，而不是直接建议安装，是对的。

可复用点：

- 插件/skill 想全局生效时，先核对 README、marketplace 路径、fork 关系是否一致。
- 只要“安装命令指向的仓库”和“用户发来的仓库”不是同一个，就应该显式提醒。

## 暴露出的失败模式

### 1. 同一 Codex 线程可能留下多份近似会话文件，日报要去重

GPT-5.6 会话在本地留下了多份同内容文件。如果按文件数直接统计，会把一次对话算成多次活跃。

### 2. 只有元数据或 IDE 日志时，不能假装恢复了 Cursor 会话内容

Cursor 当天确实活跃，但没有拿到可靠正文。这里最重要的是承认“不可恢复”，而不是用窗口标题或文件焦点反推具体问答。

### 3. 跨文档改规则时，最容易漏的是派生文档与审核稿

如果没有单独的决策日志和最后的关键词检查，很容易只改主案，不改配置表或审核文档。

## 可复用做法

### 时效信息问答模板

> 我先确认你指的是哪个产品/版本，再按截至今天的官方资料核对事实，最后给结论和对比。

### 多文档规则同步模板

> 先把用户新拍板的规则整理成一张决策表，再同步主案、派生文档和审核稿，最后跑一次交叉引用/关键词检查。

### 视觉原型迭代模板

> 先拆实现子任务，再在每轮改动后补截图验证；把“看起来不对”转成具体镜头、尺寸和接缝条件。

### 第三方插件信任模板

> 在给出全局安装命令前，先核对仓库所有者、fork 关系和 marketplace 指向，确认安装源与用户看到的是同一个。
