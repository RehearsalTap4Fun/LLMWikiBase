## 关键经验

### 1. 改图任务要写“目标视窗 + 必须保留的世界信息”，不要只说“裁成两屏”

Codex 那条会话最有效的地方，是把需求写成了精确的重构 prompt，而不是模糊地让模型“切一下图”:

- 目标尺寸直接给到 `364×644`。
- 明确是“upper swipe screen / lower swipe screen”。
- 把必须保留的对象列全: 环形营地、发光主塔、城门连接路、防御阵、底部黑色漩涡。
- 额外强调“不要简单裁边，要在窄竖屏里自然重构构图”。

可复用点:

- 视觉改图时先定义最终 viewport，而不是只定义源图尺寸。
- prompt 里同时写“保留哪些对象”和“边界如何衔接”，能明显减少模型只会硬裁切的倾向。

### 2. 图像问题不能只看原图，必须看“运行时实际渲染”

TDGame 那条长链路里，真正把问题说清的节点，不是再次调图片，而是先测出运行时场景区是 `362×1284`，而背景图是 1:2 比例；在 `cover` 下实际只露出中间约 56%，导致两侧房间被浏览器裁掉，而不是源图本身裁坏了。

可复用点:

- 遇到“看起来被裁掉太多”这类反馈，优先测容器尺寸、背景铺法和实际截图。
- 把“文件裁切问题”和“运行时 cover 裁切问题”拆开，能避免盲目反复重生图。
- 对移动长场景，单张长图全局 `cover` 往往不稳定；拆成“每个街区各自 cover 自己那一屏”更抗窗口高度波动。

### 3. 视觉迭代里，先做定量标定再修图，比凭肉眼来回试更快

Claude Code 在 TDGame 会话里多次用了网格标定图、Pillow 重排、Playwright 实际渲染截图，节奏是:

1. 看原图尺寸与比例。
2. 叠网格定位城墙、营地、防御阵、漩涡。
3. 按目标屏幕结构拼图或删接缝。
4. 打开真实页面截图验证。
5. 再回头同步改建筑位锚点。

可复用点:

- 视觉资产一旦要和交互容器、建筑位、热区对齐，就不要只停留在图片编辑器里。
- “图片重排 + 页面截图 + 锚点联调”要视为一个完整任务，而不是三件分开的事。

### 4. 数据复盘要先复用旧口径，再验证“今天的数据和字段真的存在”

datain 那条会话没有直接重写 SQL，而是先读前一天留下的 Rmd、输出目录和项目 memory，先复用口径，再确认 07-14 的数据是否真的入仓。这一步避免了“重查一遍但口径漂了”的问题。

可复用点:

- 日复盘任务先找前一天的报告、SQL、输出目录，优先延续原口径。
- 在“查指标”之前先验证新日期数据是否可用，别默认第二天就一定完整。

### 5. 真正有价值的不是跑出数字，而是把隐性字段结构挖清楚

这次 datain 最值得沉淀的事实，不是“L10 是难度墙”，而是确认了 `worldcup_2026_level_settle` 的 payload 在 `attribute1` 里，而且是双层 JSON，需要二次解析才能拿到 `level_id` 和 `result`。这个发现被写回了项目 memory，后续复查同活动时会直接节省一轮排障。

可复用点:

- 一旦发现“事件字段埋在哪、需要几层解析”这种结构性事实，立刻存进项目 memory。
- 这类事实的复用价值通常高于单次报表结论。

### 6. “难度墙”与“流失点”必须拆开判，不然结论会偏

datain 会话里最关键的修正，是不再只看“停留最大关卡分布”，而是把“最大停留点”“关卡胜率”“在前沿关主动失败人数”叠起来看。结果发现 L1 虽然堆积最多，但胜率高达 93.8%，更像流失/引导问题；真正的难度墙在 L4 和 L10。

可复用点:

- 看卡关分布时至少同时拉三类数据: 停留分布、胜率、前沿失败人数。
- “堆积多”不自动等于“难度高”。

### 7. 报告产出链路要自带排版自检，而不是生成完就停

这次 Claude Code 在 datain 里不只生成了合并版 Rmd 和 HTML，还用 headless Chrome 截图检查标题和图例重叠，发现问题后又回改图例位置再渲染一次。这个闭环很值。

可复用点:

- 让 AI 产出报告时，最好把“渲染 + 截图 QA”也包含进流程。
- 只验证“文件生成成功”不够，排版错误同样会削弱复用价值。

### 8. 后台开发服务如果被外部环境反复杀掉，就别继续强顶

TDGame 会话里 Vite 服务被连续两次外部终止。Claude 最后给出的处理是停止自动重启，转而建议在用户自己的终端里跑 `npm run dev`。这是对的，因为问题已经不在命令本身，而在后台任务生命周期不受助手控制。

可复用点:

- 连续两次启动成功但进程立刻被外部杀掉，就不要继续把“自动重启”当解决方案。
- 对需要常驻的进程，用户自管终端通常比助手后台任务更稳。

## 暴露出的失败模式

### 1. Cursor 只有运行日志时，不应硬猜使用内容

这一天 Cursor 能证明“开着且活跃”，但没有可靠 conversation summary。对这种来源，最多记“有活跃痕迹”，不要反推出 prompt 或经验。

### 2. 单张长背景图在响应式窗口里很容易漂

只要容器高度可变，整张长图全局 `cover` 就会让构图随窗口高度飘移。当天 TDGame 后半段真正稳定下来的方案，是半屏拆图 + 各自 `cover`。

### 3. 单看停留分布容易把流失错判成难度

如果没有补胜率和失败人数，L1 这种“高胜率但高停留”的关卡很容易被误判成难度墙。

## 可复用做法

### 视窗先行的改图模板

> 先定义最终可视区域尺寸、屏幕分段和必须保留对象，再要求模型“重构构图”而非“简单裁切”。

### 运行时校验模板

> 图像问题先量容器尺寸、看铺图方式、截真实页面，再决定该修素材还是修 CSS。

### 日复盘分析模板

> 先读昨天的报告和 memory，验证今天数据到位，再沿用同口径扩展，不要每次从头写 SQL。

### 隐性结构沉淀模板

> 发现字段埋藏位置、解析方式、事件表坑点后，立刻写回项目 memory，优先保存这类“结构事实”。

### 常驻进程止损模板

> 同一服务被外部杀掉两次后，停止助手侧自动拉起，切换到用户自管终端，避免反复空转。
