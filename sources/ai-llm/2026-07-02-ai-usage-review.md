## 高价值做法

### 1. Prompt 里同时给“精确主文件 + 知识域 + 指定 skill + 目标产物”，生成质量明显更稳

这次最有效的 prompt 结构不是抽象地说“帮我写开发文档”，而是四件事一次说清：

- 主输入文件是哪个绝对路径。
- 允许补充的语料范围是“知识库 K1 相关知识”。
- 明确要求走 `mindtodoc` skill。
- 目标是“生成开发文档”，不是做摘要或 review。

这个组合的价值在于，Agent 不需要猜“拿什么做依据”“要产出什么形态”，会先按 skill 约定补齐边界、模板、参考页，再落正文。

### 2. 让 skill 决定取材顺序，比让模型自由发挥更可控

会话里实际发生的顺序是：

- 先读 `using-superpowers`。
- 再定位并读取 `mindtodoc` 的 `SKILL.md`、引用规范和模板。
- 然后回读知识库里的主策划页、K1 关联页和高兼容风险规范。
- 最后才生成三份文档并做验收。

这个顺序的关键收益是：先收敛“文档边界和输出骨架”，再收敛“领域事实”。对任何依赖本地 skill 的内容生产任务，这都是值得复用的模式。

### 3. 缺失来源不要硬补脑，应该显式列缺口并限定替代依据

这次源稿里引用了几篇知识库页面，但实际路径下找不到：

- `wiki/work/k1-kvk-burning-expedition.md`
- `k1-live-activity-patterns-2026h1.md`
- `k1-new-player-retention-roi.md`

有效做法不是默认“模型自己知道这些内容”，而是：

- 明说哪些引用没找到。
- 只使用实际可读的主稿、K1 公共配置表清单、高兼容风险规范做补充。
- 在最终答复里保留这个事实，避免用户误以为所有引用都被完整消费。

这是很实用的护栏。以后凡是“基于 A 文档并参考 B/C/D”这类任务，都值得固定加一句：`如有引用源缺失，请列出缺失项和你实际使用的替代依据。`

### 4. 文档类任务也要有显式验收，不能只看“文件生成了”

这次最值得保留的不是生成动作本身，而是收尾验证拆成了几条可执行检查：

- 文件存在：3 个目标 markdown 必须都落盘。
- 章节完整：主案和配置表文档都用 `rg` 检查关键标题。
- 泄漏检查：不能把 `wiki/`、`KnowledgeBase`、`[[wikilink]]`、`source:` 之类草案痕迹带进正文。
- 占位符检查：确认 `{{日期}}`、`{{功能名}}` 等模板残留没有漏进主案/配置表。

这比“我看起来写完了”强很多。文档生成类任务完全可以像代码任务一样有验收门槛。

### 5. 模板残留要区分“阻塞缺陷”与“受控占位”

验收里确实扫到了一个残留：界面标注模板中的 `待填写`。但会话没有把它一概判为失败，而是进一步判断：

- 它只出现在单屏界面标注模板。
- 不影响主开发文档和配置表结构文档的可交付性。
- 已在答复里显式说明其性质。

这个处理很实用。以后检查生成物时，不必把所有占位符等同于阻塞，但必须说明它出现在哪、为什么可接受、影响范围是什么。

### 6. 自动化提交时要优先“缩小提交面”，而不是为了 pull/rebase 去动无关改动

同一天较早的自动化线程里，仓库存在与本次无关的未暂存改动。有效处理不是通用 `stash` 或顺手全加，而是：

- 只 `git add` 本次新增的 `ai-llm` 复盘文件。
- 检查 staged 集合和 diff stat。
- 若 `pull --rebase` 会被无关改动阻塞，就先 `fetch` 看 ahead/behind，再决定是否可直接 push 当前提交。

这对持续型知识库自动化很重要：自动化线程必须尽量避免“为了完成自己的提交，顺手处理用户其他未提交工作”。

## 暴露出的失败模式

### 1. 知识库路径和草案语法很容易漏进最终文档

如果不专门搜 `wiki/`、`KnowledgeBase`、`[[...]]`、`source:` 这类标记，生成文档很容易带出上游整理痕迹。这个问题在本地知识库驱动的工作流里尤其常见。

### 2. “引用了知识库”不等于“所有引用页都真实存在”

只要 prompt 里说了“结合知识库相关知识”，模型就很容易默认关联页都可读。实际上一旦有缺页，最安全的行为应该是显式降级，而不是自行补齐。

### 3. 跨工具日志的可恢复性差异很大，自动化不能假装覆盖完整

Codex 的 `jsonl` 足够丰富，能还原 prompt、工具调用和最终答复。Cursor 当前只有稀薄的代码跟踪库。Claude Code 本地样本又停在更早日期。对这类“全设备 AI 使用复盘”自动化，最重要的是把覆盖缺口讲清楚，而不是追求表面上的“全平台总结”。

## 可复用的 prompt 写法

### 知识库驱动文档生成

> 以 `绝对路径主文件` 为主依据，结合 `限定域知识库`，使用 `指定 skill` 生成 `明确产物类型`。如果引用源缺失，请列出缺失项，并说明你实际采用的替代依据。

### 文档生成后的验收

> 生成后不要只汇报“已完成”。请用实际命令验证：文件存在、关键章节齐全、没有知识库路径/草案标记泄漏、没有主文档级模板占位残留，并报告任何受控残留项。

### 自动化线程的安全提交

> 只 stage 本次新增或修改的目标文件。若仓库存在无关未暂存改动，不要自动 stash 或扩大提交面；先检查 staged 集合，再决定是否需要 fetch、rebase 或直接 push。
