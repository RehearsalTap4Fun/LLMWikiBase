## 高价值做法

### 1. 排查类任务先锁定“错误类型”，再横扫同类

这天最扎实的一条模式来自“切片点位错误”排查。做法不是直接改点球点位，而是先把问题拆成三层：

1. 已知点球错误是否能在当前交付物里复现。
2. 错的是源数据、生成逻辑，还是校验覆盖不足。
3. 把同类切片一起扫掉，包括点位越界、敌方半场约束、静态越位、攻防脱节、孤立球员。

这种 prompt 很有效，因为它天然逼着 Agent 做“局部复现 + 批量扫描 + 根因分类”，而不是只修单点。

### 2. 长链路需求要尽早把落地结构说成“表结构约束”

“整理试训与引导步骤需求”一开始是需求整理，后面逐步收敛成：

- 需要细化到对话文本、手势、等待、存档节点。
- 要合成单个配置表。
- 必须写进已有工作簿。
- 必须遵循 K1 规则。
- 一个“对话+手势”的概念步骤，开发落地时要拆成两个顺序步骤。

一旦把需求转成“表头、字段、顺序步骤、引用完整性、行列数、表头规则”这些硬约束，Agent 就更容易持续自检，也更容易做脚本化验证。

### 3. 自动化定义里提前写死边界，能显著减少后续空跑

“每日 AI 使用经验沉淀”这条自动化之所以描述清楚，是因为在创建前先把这些边界问明白了：

- 跑在哪个仓库。
- 覆盖哪些 AI 工具。
- 固定什么时间运行。
- 输出到哪里。
- 只有有沉淀内容时才写日志、提交、推送。

这种“先问边界，再建自动化”的方式，明显比先搭一个泛化任务再返工稳定。

### 4. 评估新工具时，把它定位成“补哪一层”比“好不好”更有用

对 `ponytail` 的判断不是泛泛地说“有帮助”，而是明确把它归到“编码 Agent 的约束/审查层”，而不是知识库架构工具。这个判断方式值得复用：

- 它替代什么。
- 它不替代什么。
- 它和现有流程是冲突、重叠还是互补。

这样做能避免把新工具一股脑塞进现有流程。

## 暴露出的摩擦点

### 环境摩擦

- 本机默认没有 `python`，多次需要回退到 `python3`。
- 知识库 `tools/search.py` 缺 `rank_bm25` 依赖，BM25 搜索直接失败。
- 表格相关任务里缺 `openpyxl`，导致已有校验/生成脚本不能直接运行。
- 一次尝试读取表格 artifact API 时，本地包也不存在。

结论很直接：这台机器上“脚本存在”不等于“脚本可用”，AI 在多仓库切换时尤其容易踩环境断裂。

### 验证摩擦

- 全量 `xlsx drift` 检查过慢，会卡住排查节奏。
- 更高效的做法是先跑能回答当前问题的窄检查，比如只比对相关 sheet、只扫关键几何约束、只输出高风险候选。

这说明验证脚本要分层：全量一致性检查适合 CI，问题排查更需要秒级、可定向的 audit 脚本。

## 可复用的 prompt 写法

### 排查配置类

> 先复现已知错误，再区分是源数据、生成逻辑还是校验覆盖问题；随后全量扫描同类配置，列出同类风险，而不是只修一个点。

### 结构化产出类

> 先按现有配置规则落地到表结构，再细化到字段级别；需要同时给出顺序步骤、引用约束和可校验项。

### 自动化类

> 每天固定时间回看前一天活跃会话，只在有 durable 内容时写日志、提交并推送；没有内容时不要制造空文件和空提交。

## 可复用实践

- 对排查类任务，优先要求 Agent 输出“错误分类框架”和“批量扫描范围”。
- 对表格/配置任务，优先要求 Agent 验证结构不变量，而不是只说“我已经改好了”。
- 对自动化任务，明确写入“无内容不提交”。
- 对新工具评估，要求 Agent 回答“放在哪一层”。
