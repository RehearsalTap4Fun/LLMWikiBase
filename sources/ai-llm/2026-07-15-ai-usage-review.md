## 关键经验

### 1. 日报/复盘类分析，第一步不是查指标，而是确认“今天的数据是不是完整的”

`datain` 会话一上来先确认 07-15 的 ODS 只到 10 点，随后把它标成“部分数据”，并顺手修正了上一版把半天数据当完整 Day2 的误判。

可复用点:

- 日维度复盘先查最新日期的入仓完整度，再决定是否参与对比。
- 半天数据必须显式标注，否则第二天复跑时很容易出现“结论反转”。

### 2. 先复用昨天的口径和产物，再扩展今天的新问题

这条分析链没有从零写 SQL，而是延续已有报告、旧口径和输出目录，在同一套口径上加“多日综合参与率”“礼包付费”“沉寂 DAC 激活”“操作模式”几个新问题。

可复用点:

- 日常追踪型任务优先复用前一版 SQL / Rmd / memory，减少口径漂移。
- 真正该新写的，是新增问题对应的补充查询，而不是把整个分析链重搭一次。

### 3. SQL 排障先修语法兼容，再继续放大分析范围

当天一个明确的排障点是 `USING` 不被当前执行环境支持，改成 `ON` 后查询恢复正常。处理方式是先让主查询跑通，再继续补“新进入 vs 回流”“逐日留存”等延伸口径。

可复用点:

- 数据分析里的报错先做最小修复，不要一边排语法一边重构整段查询。
- 兼容性问题一旦确认，后续同项目直接默认用更保守的写法。

### 4. “难度墙”不能只看堆积，还要叠胜率和前沿失败人数

这天的结论不是简单说某关人数多，而是把逐日胜率、闯关深度、中位最远关和卡关人数一起看，最后稳定识别出 L4、L10 才是真难度墙。

可复用点:

- 停留分布只说明“人堆在这里”，不等于“这里难”。
- 至少同时拉三类信息: 堆积分布、胜率、失败人数。

### 5. 报告生成最好自带“渲染后截图 QA”

当天多次把报告落成 HTML 后再回头检查图例、摘要卡片和明细表高亮，甚至因为 `DT::styleRow` 不存在而换成 `styleEqual`。

可复用点:

- 报告任务不要把“文件生成成功”当完成标准。
- 生成后截图或渲染检查一次，能早点暴露图例重叠、组件 API 误用、强调样式失效等问题。

### 6. 图像/UI 对齐问题先量运行时几何，不要急着重画素材

`TDGame` 会话里，第一次关键转折不是继续抠图，而是先确认格子位置由 `islandPlatformLayout` 驱动，再测 UI 房间格子和背景工事的实际偏差。

可复用点:

- 视觉错位先拆成两层: “源素材裁切是否对”和“运行时渲染是否漂”。
- 优先测容器、锚点、布局表、背景铺法，往往比重生一张图更快定位问题。

### 7. 一旦反复靠肉眼调图，就该升级成自动校准循环

这天最值得保留的方法，是把“还是对不上”变成一个有收敛条件的 loop：脚本实测渲染框，按目标裁切框反推布局，写回代码，再重测，直到每格误差 < 2px。

可复用点:

- 视觉对齐要尽量从“主观截图判断”升级成“有误差阈值的闭环校准”。
- Playwright 实测 + 目标坐标反推 + 自动回写，是前端/游戏 UI 对位问题里非常强的通用套路。

### 8. 响应式背景图如果依赖 `center / cover`，窗口高度一变就会漂

会话中明确识别出“你的窗口和我校准时不一样”才是错位根源，因为背景图会在矮窗口里被上下裁切，而 UI 锚点没有同步变化。后续修法改成更稳定的锚定方式和像素定位。

可复用点:

- 只在单一窗口高度校准通过，不代表方案稳定。
- 需要跨窗口稳定时，要把背景锚定策略和 UI 锚点一起设计，而不是只调某组百分比。

### 9. 状态切换也会破坏对齐，必须测“开战前后”的真实位移

最后一个 bug 不是静态页面里的错位，而是战斗开始后 HUD 插入布局流把格子整体挤下去。真正修复不是继续调位置，而是定位到 DOM/布局变化源。

可复用点:

- 视觉对齐不要只验静态态，还要验关键状态切换。
- “开战后偏移”这类问题，本质上常常是 reflow / DOM 插入，不是素材问题。

## 暴露出的失败模式

### 1. Cursor 只有运行日志时，不应硬猜对话内容

- 进程活跃只能证明 IDE 在开，不能证明具体聊了什么。

### 2. 同一个视觉问题靠截图往返太多次，说明缺少量化闭环

- 当用户已经多次反馈“还是不对”，继续肉眼微调通常收益很低。

### 3. 把“当天数据”默认当完整数据，是日复盘最危险的口径坑

- 这类坑不会让 SQL 报错，但会直接把结论带偏。

## 可复用做法

### 日复盘起手模板

> 先确认新日期数据是否完整，再复用昨天的 SQL / memory / 报告结构扩展分析。

### 报告 QA 模板

> 生成报告后立即截图或渲染检查一次，把样式/组件问题当成正式 bug 处理。

### 视觉对齐闭环模板

> 先实测运行时几何，再用脚本“测量 → 求解 → 回写 → 复测”，直到误差收敛到阈值内。

### 状态切换验收模板

> 对齐通过后，再补测关键状态切换前后的位置差，确认不是动态布局把元素挤偏。
