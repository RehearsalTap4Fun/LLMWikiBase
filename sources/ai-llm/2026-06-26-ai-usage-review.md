## 高价值做法

### 1. 前一轮已经把边界和验证讲清后，续跑 prompt 可以极短

这次用户在原线程里只发了：

> 按照建议修复

之所以这句足够，是因为线程前文已经沉淀了三样东西：

- 具体异常样例：`1011`、`1013`、`1019`、`6001`、`6002`
- 已有约束：任意球人墙、角球接应、队友间距不能回退
- 可执行验证：坐标测试、一致性脚本、生成后的 xlsx 复扫

结论：如果前一轮已经把“要改什么”和“不能改坏什么”编码进线程，续跑时不需要重写长 prompt，直接让 Agent 落地即可。

### 2. 单点 bug 值得上升成“同类场景归一化”，但前提是保住旧护栏

这次不是只修一个 preset，而是把同类“可接应但没人接应”的问题一起处理：

- `tap_in` / `rebound`：把接应球员推到门前或二点落点附近
- `throw_in`：把第一接应球员贴到 `TargetPoint`
- 同时保留此前已建立的 free kick wall / corner receiver 几何约束

这类扩写有效，因为它不是顺手扩需求，而是围绕同一失败模式补一层“可玩站位归一化”。

### 3. 最稳的验证链还是“三层验证”

这次最终闭环比较完整：

- 局部测试：全量坐标/站位测试 12 项通过
- 全局规则：`scripts/check_preset_consistency.py` 报 `79 preset 全部合规`
- 产物验证：重新生成 xlsx 后，复扫“优先接应问题”结果为 `0`

这个顺序很值得复用。局部测试回答“修的是不是这类 bug”，一致性脚本回答“有没有把别的 preset 带坏”，产物复扫回答“真实交付物现在是否可用”。

### 4. 脏工作区里，明确不碰无关改动本身就是好习惯

最终输出里明确说明仍有一个无关的未跟踪 markdown 文件未处理。这个习惯很重要：在配置生成或产物重导出类任务里，Agent 很容易顺手吞掉旁支改动；显式声明“不处理无关变更”能降低误提交风险。

## 暴露出的失败模式

### 1. 只按 transcript 目录日期找会话会漏掉跨日续跑

这条高信号会话的 transcript 存在 `2026/06/24/` 目录下，但 2026-06-26 仍然活跃。真正可靠的是：

- `codex-hook-events.ndjson` 里的 `UserPromptSubmit` / `Stop` 时间
- transcript 内部的事件时间戳

对这类日复盘自动化来说，hook 时间线比“文件在哪一天的目录里”更重要。

### 2. ambient suggestion 会话噪声很大，不宜直接当个人经验样本

2026-06-26 还出现了多条系统 suggestion prompt，会话没有稳定 transcript path，但输出里确实给出过一个有用信号：KnowledgeBase 的 `tools/search.py` / `tools/lint.py` 运行环境可能缺 `rank_bm25`、`pyyaml`。

这个信息可以当低置信运营提示，但不应和真实用户工作线程同权重对待。

## 可复用的 prompt 写法

> 沿用上一轮确认过的规则和验证，直接按建议修复；不要扩大范围，保留既有几何约束，修完后给我测试结果和产物复扫结果。
