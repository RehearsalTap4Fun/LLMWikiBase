## 关键经验

### 1. 用户说“某关卡行为不对”时，先查关卡配置图，不要先怀疑引擎

`Mechanic` 第一波“所有怪都直达核心、不分叉”的根因不是移动逻辑，而是：

- `stage_mechanic` 第一波固定从 `mid` 出怪。
- `mid -> slot_1_2` 当时只有单出口链路。

这说明对关卡型问题，排查顺序应该先是：

1. 当前 stage 的 spawn 配置。
2. 入口房和 `next` 连通图。
3. 再看引擎是否正确消费这些配置。

如果用户能直接给出“关卡名 + 第一波 + 从哪个入口进 + 预期应该分到哪里”，定位速度会明显快于泛化描述“分叉坏了”。

### 2. 单元模拟通过不代表用户看到的页面就对，验证要分层升级

TDGame 这天最有价值的模式，是把验证从“逻辑正确”升级成“三层证据”：

- 配置/模拟层：内容测试和 sim 测试先确认连通图、分叉目标和时序逻辑。
- App 层：用 fake timer 跑真实 `Mechanic -> Start Run` 交互，断言 DOM 中出现右路目标。
- 浏览器 E2E 层：在真实页面点击、等待、截图，并检查右路怪物是否真的可见。

这比反复改引擎稳得多。尤其当用户反馈“我页面上就是没看到”时，最佳做法不是继续猜，而是把问题升级到更接近用户视角的验证层。

### 3. 视觉 bug 要翻译成几何/层级约束，而不是只凭截图感受调 CSS

当天多个 UI 问题的修法都体现了这个思路：

- 上下石板路“看起来没连上”，实际是纵向路径用 `width + rotate(90deg)`，导致高度计算错了。
- 怪物被房间 UI 遮挡，根因是怪物层挂在房间内部，应该升到全局 `EnemyLayer`。
- 房间开口和码头是否合理，不应该靠手写 DOM，而应从入口房和方向规则动态生成。

经验是：收到“看起来不对”的反馈后，先把它改写成几何、层级或数据生成规则，再动样式。这样修出来的东西才可测试、可复用。

### 4. 真实调试里，时间分布本身就是 bug 线索

右路“理论上存在但用户早期看不到”的那次，不是路由缺失，而是第三只怪才会去右侧，导致前几秒观察像“没有右路”。最后的解决不是随机化，而是把三出口策略改成稳定的“左、右、中”外侧优先轮转，并补时间窗口测试。

这类问题说明：

- “最终会发生”不等于“用户会及时看到”。
- 对体验验证，要写带时间窗口的断言，而不是只验证最终集合里包含某目标。

### 5. 两套测试框架并存时，要尽早隔离职责

补 E2E 后暴露出一个很实际的问题：Vitest 把 Playwright spec 也扫进去了。最终修法是让 `tests/e2e` 只由 `npm run test:e2e` 执行，单元测试 runner 明确排除这类文件。

这个经验很通用：

- 单元/组件测试和 E2E 可以共存。
- 但发现“一个 runner 开始执行另一套 spec”时，要立刻修测试边界，不然后续 CI 会越来越脆。

### 6. 对 Codex 会话管理，要把“消失”优先理解为“被归档”

`Command + Shift + A` 之后会话“消失”的问题，最终证据表明线程还在，只是被归档/隐藏。恢复时依赖的不是猜测，而是：

- `state_5.sqlite` 里按 `updated_at` 查线程。
- 结合 thread id、cwd、最近预览内容确认是目标线程。
- 再取消归档。

这条很适合长期保留：以后遇到 Codex 会话“找不到”，先查归档状态和线程索引，不要先假设数据丢失。

## 暴露出的失败模式

### 1. 过早相信“默认关卡逻辑”会掩盖 stage-specific 配置问题

如果只看全局移动规则，而不看 `stage_mechanic` 的具体入口和连通图，会把配置问题误判成引擎 bug。

### 2. 把“逻辑存在”误当成“用户可见”

右路怪物、纵向石板路、怪物移动层这些问题都说明：内部状态正确，不代表用户视角正确。以后这类问题要更早进入 App/E2E 验证。

### 3. 有客户端本地文件，不等于当天有可复盘会话

Cursor 还有本地数据库，Claude Code 也还有历史文件，但 2026-07-08 并没有新的可恢复会话。跨客户端日报仍要坚持“只记当天可证实的活跃会话”。

## 可复用做法

### 关卡型 bug 的提问模板

> 指明关卡名、波次、入口、当前观察结果、预期路线，最好附上当前页面 URL 或截图。这样 AI 可以先查配置图，再决定是否需要动引擎。

### 分层验证模板

> 先配置信号与单测，再 App 级真实交互，最后浏览器 E2E 和截图。不要在用户说“页面上没看到”时只停留在模拟层。

### 视觉问题的排查模板

> 把截图反馈转写成“几何长度、方向计算、z-index、挂载层级、数据生成规则”之一，再改代码；避免纯感性调样式。

### Codex 会话恢复模板

> 先查线程索引和归档状态，再按 thread id、cwd、最近内容确认目标线程，最后恢复显示。把“误归档”视为第一嫌疑，不把“找不到”直接等同于“删除”。
