## 关键经验

### 1. 先把“长期上下文入口”配好，后续会话质量会明显更稳

当天最值的一步不是立刻做业务分析，而是先把 K1 的本地仓库入口写进全局 `~/.claude/CLAUDE.md`。这样后面的任何新会话都知道去哪里查客户端、服务端和配表，不用每次重新解释项目结构。

可复用点:

- 需要跨项目长期复用的信息，优先写进全局上下文，而不是散落在单次对话里。
- 记录本地路径时，顺带写技术栈、主要目录和当前分支，能减少后续“先认路”的上下文消耗。

### 2. 对代码/配置类助手，本地路径比远端地址有用得多

当天明确验证了一件事: 对 AI 助手来说，远端仓库地址更像“来源记录”，本地仓库路径才是真正高频使用入口。因为搜索、读文件、查 git 历史、定位调用链，几乎都依赖本地工作区。

可复用点:

- 把“本地路径 + 远端地址”同时记录，但默认让助手优先走本地。
- 若本地仓库尚未完整 checkout，要显式标注这个状态，避免后续误判“文件不存在”。

### 3. 技能安装不是终点，安装后立刻做“完整性验收”

`datain-skill`、`igame-skill` 刚装好时都暴露出同类问题: 安装结果里只有 `SKILL.md`，缺少它引用的 `scripts/` 和 `references/`。`ai-to-sql` 更进一步，第一次还被装进了错误的嵌套目录。真正有效的做法不是相信“安装成功”提示，而是马上检查安装目录与远端仓库文件清单。

可复用点:

- 装完 skill 后立刻 `find` 一次本地目录，并与远端仓库文件清单对照。
- 只要 skill 依赖脚本或 references，就不能只看 `SKILL.md` 是否存在。
- 遇到错误嵌套路径时，直接手动挪正目录、补全文件并重建 symlink，比反复重装更快。

### 4. 数据分析链路要按“接入 -> 鉴权 -> 埋点存在性 -> 平台可查性”逐层排

datain 会话里最稳的路径不是上来就问“为什么查不到数据”，而是先装技能、跑鉴权、再去 `game` 仓库确认 `worldcup_2026_*` 埋点是否真实存在。最后才得出关键结论: 代码里已有上报，不代表 datain 聚合层已经建好可查指标；“查不到”可能是发布未到线上，也可能是数据建模还没补。

可复用点:

- 先确认埋点是否真的在代码里，而不是默认文档写的接入方式一定准确。
- 区分“原始事件存在”和“平台指标可查”这两个层次。
- 当平台查不到时，把问题拆成“部署状态”与“建模状态”分别追。

### 5. 自然语言转 SQL 最好默认“先生成，不自动执行”

`ai-to-sql` 的技能说明很值得保留: 默认只返回 SQL，只有用户明确要求执行才真正跑查询。这个约束很适合内部数仓场景，因为它把“SQL 质量校验”和“数据读取动作”分开了，既方便审查，也减少误查误跑。

可复用点:

- 对高成本或高风险查询，先生成 SQL，再决定是否执行。
- 一旦确认要执行，最好把结果物固定输出到项目目录，而不是留在 `Downloads`。

### 6. MCP 降摩擦最有效的做法，是把“启用 server”和“工具权限”分开配置

datain 项目里已经有 `enableAllProjectMcpServers: true`，说明“新会话自动启用 `.mcp.json` 里的 server”其实早就生效了。真正还会打断流程的是第一次调用工具时的权限确认，所以后面又在 `permissions.allow` 里加了 `mcp__dingtalk-sheet` 与 `mcp__dingtalk-docs`。

可复用点:

- `enableAllProjectMcpServers` 解决的是“是否启用 server”。
- `permissions.allow` 解决的是“调用具体工具时还要不要再问”。
- 两者都放开后效率很高，但写入类工具也会直接执行，必须只给自己真正信任的 server。

### 7. MCP / CLI 配置里，标识名要优先考虑工具约束，而不是人类命名习惯

给 Claude 加 `钉钉文档` MCP 时，第一次失败不是网络或鉴权问题，而是 CLI 不接受中文名称。改成 `dingtalk-docs` 后立即成功。这种问题很琐碎，但很常见，应该尽量在第一次命名时就避开。

可复用点:

- MCP server 名称统一用 ASCII、短横线或下划线。
- 人类可读的中文名可以写在说明里，不要直接拿去当 CLI 标识。
- 新增 MCP 后记得提醒“需新开会话/重启后生效”。

### 8. 让 AI 产出可复用分析物，而不是只给口头结论

datain 会话后半段的价值，不只是分析出“第 1 关是最大卡点”，而是顺带生成了 `report.Rmd`、HTML 和截图验证流程。这样后续复盘、转发和二次修改都有落点。

可复用点:

- 让 AI 把分析写成项目内可保存的报告，而不是只回聊天结论。
- 报告生成后补一次渲染/截图验证，避免只停留在“文件写出来了”。

### 9. 从 AI 生成切回人工生图时，最重要的是把资产约束说清

TDGame 的背景图交接非常干净: 助手没有只给一个目录，而是把“统一背景图路径、1024×2048 尺寸、上半经营/下半地牢、CSS 按 50%/50% 切屏”一起交代清楚。这让人工重生图变成一个明确任务，而不是重新猜 UI 结构。

可复用点:

- 交接视觉资产时，至少给路径、尺寸、布局比例、引用位置。
- 如果多个界面共用同一张图，要明确说明它是连续图还是独立图。

### 10. 任务建议类提示词，要把“为什么是现在”写进约束里

Codex 当天那组建议生成 prompt 有个很实用的结构: 不只要求“给建议”，而是强制检查 `Why this user / Why now / Why Codex / Why not already handled`。这会显著减少泛泛而谈的建议。随后再单独跑一次安全过滤，把合规判断从“生成阶段”拆出去。

可复用点:

- 对“给我推荐接下来做什么”这类任务，显式要求时间性和可执行性。
- 生成与合规筛选分成两步，比把所有约束塞进一次生成更稳。

## 暴露出的失败模式

### 1. Cursor 运行了，不等于有可复盘正文

当天 Cursor 本地库和 tracking DB 都没给出可信对话正文，只有进程级运行数据。这里最重要的是承认来源不够，而不是硬从运行日志猜测使用内容。

### 2. “安装成功”提示不代表工具链真的可用

`skills add` 至少暴露了两类坑: 只拉到 `SKILL.md`，以及安装路径嵌套错误。任何需要脚本和 references 的 skill，都必须验收。

### 3. 文档说法和真实接入机制可能早就脱节

`worldcup_2026` 埋点检查里，文档还在说 `bi.xml`，真实实现已经是 Go 常量注册。只信文档、不看代码，会把排障方向带偏。

## 可复用做法

### 全局上下文配置模板

> 先把本地仓库路径、技术栈、关键目录、当前分支写进全局上下文文件，让后续新会话无需重复认路。

### skill 安装验收模板

> 安装完成后立刻检查本地目录文件清单，并与远端仓库对照；只要缺 scripts/references，就不要进入正式使用阶段。

### 数据分析排障模板

> 先确认接入和鉴权，再核实代码埋点是否存在，最后区分“原始日志存在”和“平台指标可查”。

### MCP 降摩擦模板

> `enableAllProjectMcpServers` 负责自动启用 server；`permissions.allow` 负责免去工具级确认，两层都配好才是真正顺滑。

### 视觉资产交接模板

> 交接时同时给出路径、尺寸、布局比例和引用位置，让人工替换素材时不需要再反推 UI 结构。
