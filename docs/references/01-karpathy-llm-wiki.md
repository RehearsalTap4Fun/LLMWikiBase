## 核心理念

这份 gist 描述了用 LLM 构建个人知识库的模式。它被定位为一份"理念文件"，意在复制粘贴给 LLM agent，传达高层概念，由 agent 与用户协作搭建具体实现。

与标准 RAG 的关键对比：RAG 在查询时检索原始文档片段（"LLM 在每个问题上都从零重新发现知识"），而本模式让 LLM"增量地构建并维护一个持久 wiki"——一组介于用户与原始素材之间的结构化、互链 markdown 文件。

核心论断："wiki 是一个持久的、不断累积的产物。" 交叉引用已经存在，矛盾已被标记，综合反映了读过的一切。作者原话："Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。" 用户负责取材、探索和提问；LLM 负责摘要、交叉引用、归档和簿记。

适用场景举例：个人追踪、研究、读书（如 Tolkien Gateway 这类 fan wiki）、商业/团队 wiki、竞品分析、行程规划。

## 三层架构

- **原始素材（Raw sources）**——精选的源文档（文章、论文、图片、数据）。不可变；"LLM 只读不改"。这是事实来源。
- **Wiki**——LLM 生成的 markdown 文件目录（摘要、实体/概念页、对比、综合）。"LLM 完全拥有这一层。"
- **Schema**——一份定义结构、约定和工作流的文档（如 CLAUDE.md 或 AGENTS.md）。它"让 LLM 成为有纪律的 wiki 维护者而非通用聊天机器人"，随时间共同演进。

## 三个操作

- **Ingest（投喂）**——投入一篇素材，让 LLM 处理：阅读、讨论要点、写摘要页、更新 index、更新实体/概念页、追加 log 条目。"单篇素材可能触及 10-15 个 wiki 页面。" 可逐篇有监督处理，也可批量投喂。
- **Query（查询）**——对 wiki 提问；LLM 找到相关页并综合出带引用的回答（markdown、对比表、Marp 幻灯片、matplotlib 图表、canvas）。关键洞见："好的回答可以作为新页面归档回 wiki"，让探索不断累积。
- **Lint（健康检查）**——定期巡检：矛盾、过期论断、孤立页、缺失的概念页、缺失的交叉引用、可由网络搜索补全的数据缺口。

## 索引与日志

- **index.md**——面向内容的目录，列出每个页面的链接、一句话摘要、可选元数据，按类别组织。回答查询时先读它。在中等规模（约 100 篇素材、数百页）下工作良好，"避免了基于 embedding 的 RAG 基础设施"。
- **log.md**——按时间倒序、追加式记录 ingest/query/lint。统一前缀使其可被 unix 工具解析——例如 `grep "^## \[" log.md | tail -5` 返回最近 5 条。

## 可选：CLI 工具

随着 wiki 增长，对 wiki 页面的搜索引擎是最显而易见的工具。作者推荐 qmd——一个本地 markdown 搜索引擎，支持 BM25/向量混合搜索 + LLM 重排，提供 CLI 与 MCP server。自己写个简单脚本也行。

## 技巧

- **Obsidian Web Clipper** 把网页文章转成 markdown，便于快速取材。
- **本地下载图片**（通过 Obsidian 附件设置 + 快捷键），让 LLM 直接引用图片。注意：LLM 无法一次读取内联图片 markdown——先读文字，再单独看图。
- **Obsidian 图谱视图** 展示 wiki 的形态（枢纽 vs 孤立页）。
- **Marp** 从 wiki 内容生成幻灯片。
- **Dataview** 对 YAML frontmatter 跑查询生成动态表格。
- wiki "只是一个 markdown 文件的 git 仓库"，免费获得版本历史、分支、协作。

## 为什么有效

知识库中繁琐的部分"不是阅读或思考——是簿记"。人类放弃 wiki 是因为维护负担超过了价值；而 LLM"不会厌倦、不会忘记更新交叉引用、能一次触及 15 个文件"。人类负责精选素材、指引分析、提好问题；LLM 干其余的活。

## Memex 渊源

该理念与 Vannevar Bush 的 Memex（1945）相关——"一个个人的、精心策划的知识库，文档之间有联想路径"。Bush 的愿景是私有、主动策划、文档间的连接与文档本身同样有价值。"他没能解决的部分是：谁来做维护。LLM 解决了这件事。"
