# LLM Wiki vs RAG 知识库

> **来源**：https://atlan.com/know/llm-wiki-vs-rag-knowledge-base/
> **作者**：Emily Winks（Atlan，2026 年 4 月更新）
> **获取日期**：2026-06-12
> **说明**：本文为 AI 转写的结构化摘要，非逐字原文。以原链接为准。

## 核心区分：编译时 vs 查询时

根本差别在于知识何时被组装：
- **LLM wiki** 把结构化 markdown 索引直接载入上下文，模型预先读取相关文章，无需向量数据库（编译时组装）。
- **RAG** 在查询发生的那一刻从向量库检索语义相关片段（查询时组装）。

文章强调这是规模假设的差别，不是智能的差别。wiki 假设知识有界且稳定；RAG 假设知识庞大、动态、多领域。

## 什么是 LLM Wiki

由 Karpathy 于 2026 年 4 月提出的、基于 markdown 的个人知识库，建立在三个文件夹上：
- **`raw/`**：非结构化源材料（PDF、笔记、网页剪藏、原始研究）
- **`wiki/`**：LLM 编译的摘要文章，每个概念/主题一篇
- **`index.md`**：所有文章的总图，大小须放进上下文窗口；是 LLM 的入口

附加组件：定期的健康检查提示（标记过期/不完整/矛盾条目）、反向链接（作为轻量知识图谱边）。关键洞见是把 LLM 当"编译器"——把原始输入综合成结构化文章，而非仅查询。

## 什么是 RAG 知识库

向量索引的文档库 + 检索层。LLM 从不载入全部语料，只在检索到的证据上落地回答。阶段：文档库 → 分块层 → embedding 模型 → 向量数据库（Pinecone、Weaviate、pgvector）→ 检索层（取 top-K）→ LLM 落地综合。局限：输出质量完全取决于上游数据质量；默认不内建访问控制、新鲜度、血缘；分块与 embedding 策略带来工程开销。

## 权衡对照表

| 维度 | LLM Wiki | RAG 知识库 |
|---|---|---|
| 知识规模 | 约 100–200 篇（索引须放进上下文） | 百万级文档，无上下文上限 |
| 基础设施 | 零（markdown 文件） | 向量库、embedding 流水线、检索层 |
| Token 效率 | 小规模下最高省 95% | 单查询成本更高；大 N 下更优 |
| 新鲜度 | 手动 + LLM 健康检查；无自动传播 | 流水线触发重索引；可近实时 |
| 访问控制 | 无（仅文件权限） | 部分；取决于实现 |
| 多用户/并发 | 无事务 DB 会有竞态、写冲突 | 为并发读设计；写治理因实现而异 |
| 搭建时间 | 数小时 | 数天到数周 |
| 最适知识类型 | 稳定、精选、个人规模 | 动态、大规模、多领域企业级 |
| 失败模式 | 上下文溢出、内容过期、无访问控制 | 分块差、embedding 漂移、数据未治理 |
| 治理 | 不内建 | 部分；取决于实现 |

## 5万–10万 Token 阈值

这是 wiki 方案停止可靠工作的临界点。阈值以下，wiki 在简单性与 token 效率上胜出；超过之后，索引无法放进上下文，上下文窗口限制迫使引入检索层（无论存储格式）。token 效率优势在与优化后的 RAG 对比时收窄，超过单个上下文窗口后消失。

## 混合 / 双层方案

二者并不互斥，三种组合：
- **Wiki 作为 RAG 之上的精选上下文**：结构化 wiki 文章作为高置信"种子"上下文，在 RAG 检索佐证前给 LLM 一个解释框架，减少幻觉。
- **双层分离**：wiki 持有"我们确知的"层（认证概念、稳定定义）；RAG 处理"语料当前有什么"层（实时搜索、广覆盖）。
- **Wiki 作为受治理的元数据层**：最强的情形是"wiki"即一个受治理的数据目录（认证、访问受控的元数据）。

## 何时用哪个

- **从 LLM wiki 起步**：个人研究、约 150 页以内的稳定语料、单人、零基础设施约束
- **从 RAG 起步**：企业规模、动态内容、多用户、监管环境、超出上下文限制的语料
- **两者都投入**：需要精选锚点 + 广检索的企业 AI agent，且源数据已受治理

## 关键结论

- 在约 5万–10万 token 以下，LLM wiki 在 token 效率上胜出；RAG 在企业规模胜出。
- 真正的企业问题不是 wiki vs RAG，而是源数据是否足够可信。
- 访问控制、新鲜度、并发是治理问题，不是检索架构问题。
- 底线："**规模决定架构，治理决定结果。**" Karpathy 明确把 wiki 限定在个人研究者范围，所以它的局限是设计取舍而非缺陷。

## 对本知识库的意义

本项目是**个人**知识库，规模几乎必然落在阈值以下——因此采用 Karpathy 式纯 LLM Wiki（方案 B：markdown + 轻量脚本）是正确选择，无需向量库/RAG。等规模超过约 100–200 页或需多人协作时，再评估演进。
